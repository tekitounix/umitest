# ハードウェアI/O処理設計 共通方針

本ディレクトリのドキュメントに共通する設計方針を記載する。

## ISRと非ISRの処理分離

タイマー割り込みやDMA完了割り込みなどのISR（割り込みサービスルーチン）内では、処理を最小限に留めなければならない。ISR内で長い処理を行うと、他の割り込みの遅延やリアルタイム性の喪失を招く。

- **ISR内で行うこと:** GPIO読み取り、DMAバッファへの書き込み、アキュムレータへの加算など、数命令で完了する処理のみ
- **ISR外で行うこと:** デバウンス判定、フィルタリング、イベント生成、状態更新などのデータ処理

ISR外の処理は、RTOSのタスク、メインループのポーリング、ソフトウェアタイマーコールバックなど、実行環境に応じた非割り込みコンテキストで行う。本ドキュメント群では、これらを総称して「**処理ループ**」と呼ぶ。

## 周期設計の方針

| I/O種別 | サンプリング周期 | 処理ループ周期 | 備考 |
|----------|-----------------|---------------|------|
| ボタン | 1kHz | 1kHz | デバウンスに必要な分解能を確保 |
| ポテンショメータ | 10kHz (ADC/DMA) | 1kHz | オーバーサンプリングでノイズ低減 |
| MIDI UART | DMA受信 (連続) | 1kHz | パース遅延がMIDI伝送遅延と同等 |
| ロータリーエンコーダー | 4kHz | 1kHz | 高速回転の取りこぼし防止に4kHz必要 |

- ボタン・ポテンショメータ・MIDI UARTの処理ループは共通の1kHz周期で統一でき、同一の周期処理として駆動できる
- エンコーダーのサンプリングのみ4kHzの独立タイマーが必要。ただしISR内の処理はLUTデコード+加算のみで極めて軽い
- エンコーダーの処理ループ（Velocity計算やUI反映）は他と同じ1kHzで統一可能
