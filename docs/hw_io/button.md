# ボタン入力処理

| 項目 | 値 |
|------|-----|
| 手法 | 低優先度タイマー割り込みでの固定周期サンプリング + デバウンス |
| 理由 | ピン変化割り込みのチャタリング問題を回避し、安定した負荷で確実に状態検出するため |

---

> 共通の設計方針（ISR/処理ループの分離、周期設計など）は [README.md](README.md) を参照。

## 入力サンプリング

- 低優先度タイマー割り込みで固定周期 (**1kHz**) でサンプリング
  - 一般的なタクトスイッチのチャタリング持続時間はデータシート上で5〜10ms程度（例: Omron B3F系 max 5ms、Alps SKQG系 max 10ms）。劣化や大型メカニカルスイッチでは10ms以上に達する場合もある。1ms周期ならチャタリング区間で5〜10サンプル以上を確保でき、デバウンス判定に十分な分解能がある
  - 人間の意図的なボタン操作（タップ）の最短押下時間は概ね50〜80ms程度。素早い連打では30ms程度まで短くなる場合もある。1kHzなら最短ケースでも30サンプル以上で取りこぼしの心配がない
  - 500Hzでも動作するが、デバウンスの時間分解能が2msに粗くなる。2kHz以上はボタン用途では過剰
  - ポテンショメータやMIDI UARTの処理ループも同じ1kHz周期で駆動できるため、共通のタイマーで統一すると設計がシンプルになる
- GPIOをPort単位で読み、そのままリングバッファに入れる。Port単位で読むことで複数ボタンを1回のレジスタアクセスで取得できる。マスクはISR内では行わず、処理ループ側で必要なビットを取り出す際に行う。ISR内の処理を最小化でき、生のポート値を保持することでデバッグや後からのピン追加にも対応しやすい
- GPIOがDMA対象のMCUではDMAでも可
- スリープ中だけピン変化割り込みを有効化しWake用に使う
- Wakeしたら改めてボタン状態を確認し、一定期間押されたままなら実際にスリープ解除する

### マトリクススキャン

ボタン数がGPIOピン数を超える場合、行列（マトリクス）配線でスキャンする。

- タイマーISR内で行（または列）を1本ずつ駆動し、対応する列（または行）をPort単位で読む
- 1回のISRで全行をスキャンするか、1行/ISRで巡回するかは行数とISR負荷で決める
  - 例: 4行 × 8列、1kHz ISRで全行スキャン → 1ms内に4回のGPIO出力切替+読み取り
  - 例: 8行 × 8列、4kHz ISRで1行/ISR → 全キー更新は500Hz（2ms周期）
- 行切替後、GPIOの安定待ち（数μs程度）が必要な場合がある。1行/ISR方式では、ISR冒頭で前回駆動した行の列を読み取り、次の行を駆動してISRを終了する。次回のISR呼び出し（数百μs後）までにGPIOは十分安定するため、ソフトウェアループやタイマーによるCPUブロックが不要になる
- ゴースト（同時押しの誤検知）対策としてダイオード付きマトリクスが望ましい。ダイオードなしの場合はソフトウェアで2キーロールオーバー制限を設ける

### 外部I/Oエキスパンダ / シフトレジスタ

GPIO数が不足する場合や、配線を簡素化したい場合に外部ICを使う。

- **シフトレジスタ（74HC165等）:** SPI/GPIO経由でシリアル読み出し。タイマーISR内でクロック+ラッチ操作し、ビット列を取得する。ボタン数が多くてもGPIOは3〜4本で済む
- **I/Oエキスパンダ（MCP23017、PCF8574等）:** I2C/SPI経由で読み出し。I2Cの場合はISR内での通信が長くなるため、ISRではフラグのみ立てて処理ループで読み出すか、DMA付きI2Cを利用する
- いずれの場合もデバウンス処理は直結GPIOと同じ方式をそのまま適用できる

## データ処理

- 処理ループで周期的にリングバッファをポーリング（サンプリングと同じ1kHz、または同期した周期）
- デバウンス処理を行う（カウンタ積分方式かシフトレジスタ多数決）
- 状態変化のイベントを生成してキューに入れる
- 同時に状態ビットマスクも共有メモリに置く

## やるべきでないこと

- **ピン変化割り込みは避ける**
  - チャタリングで割り込み嵐になりやすく、低負荷のつもりが最悪時に最悪になる
  - ノイズや ESD でも同様に割り込みが増える
  - 割り込み側でデバウンスすると結局タイマーやロックアウトが必要になり、設計が複雑化しやすい

## デバウンス

### カウンタ積分方式

- raw が押下側なら cnt を増やす、離脱側なら減らす
- cnt が上限に達したら pressed と確定、下限なら released と確定
- 閾値と上限でデバウンス時間が決まる

**利点:**
- 乗算不要で軽い
- ノイズが混ざっても自然に吸収できる

**目安:**
- サンプル周期 1ms から 2ms
- 確定まで 5ms から 20ms 程度
- データシート上のチャタリング持続時間は製品により5〜10ms程度（劣化時はそれ以上）。安全マージンを含めて10〜20msで確定させる設計が多い
- 製品のスイッチ品質で詰める

### シフトレジスタ多数決方式

履歴をビットシフトし、過去N回すべて「1」ならON、すべて「0」ならOFFと判定する積分型ロジック。N=4、サンプル周期1msなら4ms連続一致で確定となる（N=8なら8ms）。

```c
// 疑似コード: 周期ハンドラ内で実行
history = (history << 1) | gpio_get_value();
if ((history & MASK) == MASK) state = ON;
else if ((history & MASK) == 0) state = OFF;
```

### 方式の使い分け

- **シフトレジスタ方式:** シンプルで省リソース（uint8_t 1個 + ビット演算のみ）。ボタン数が多い場合やリソースが限られる場合に適する
- **カウンタ積分方式:** ノイズの多い環境で柔軟にチューニングしやすい。閾値の上下で非対称なデバウンス時間（押下を速く、離脱を遅くなど）を設定できる
