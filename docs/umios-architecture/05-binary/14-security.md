# 14 — セキュリティと暗号

## 概要

アプリ署名検証、OTA 完全性検証、暗号プリミティブ。
本章は UMI-OS のセキュリティモデルと暗号ライブラリの設計を規定する。

| 項目 | 状態 |
|------|------|
| CRC32 完全性検証 | 実装済み |
| Ed25519 署名検証 | 新設計 |
| SHA-256 / SHA-512 | 新設計 |
| 鍵管理 | 新設計 |
| OTA/DFU 検証フロー | 新設計 |

---

## セキュリティモデル

### 何を守るか

| 脅威 | 対策 | 検証タイミング |
|------|------|--------------|
| 改竄された .umia の実行 | Ed25519 署名検証 | Loader がアプリロード時 |
| OTA 転送中のデータ破損 | CRC32 + Ed25519 | Updater が FW_VERIFY 時 |
| 不正な OTA の適用 | Ed25519 署名検証 | Updater が FW_COMMIT 前 |
| ロールバック攻撃 | バージョン番号の単調増加チェック | Loader / Updater |

### ターゲット別の適用

| ターゲット | 署名検証 | 備考 |
|-----------|---------|------|
| AppTarget::RELEASE | **必須** | 署名不一致でロード拒否 |
| AppTarget::DEVELOPMENT | スキップ | 開発用。署名フィールドはゼロ |
| AppTarget::USER | スキップ | ユーザー自作アプリ |

---

## 暗号プリミティブ

### 一覧

| アルゴリズム | 用途 | 出力サイズ |
|-------------|------|-----------|
| SHA-256 | ハッシュ（汎用） | 32B |
| SHA-512 | Ed25519 内部で使用 | 64B |
| Ed25519 | デジタル署名（検証のみ） | 64B 署名 / 32B 公開鍵 |
| CRC32 | 転送完全性チェック（IEEE 802.3） | 4B |

### 設計方針

- **検証のみ**。デバイス上で署名を生成する機能は不要（署名はビルド時にホスト PC で行う）
- ヒープ割り当て不可。スタック上のバッファで完結する
- `process()` パス（リアルタイム）では呼ばない。Loader / Updater（SystemTask）でのみ使用
- 純粋な C++ 実装（アセンブラ最適化は将来のオプション）

### ROM/RAM フットプリント概算

| コンポーネント | ROM | RAM (スタック) |
|--------------|-----|---------------|
| SHA-256 | ~2KB | ~200B |
| SHA-512 | ~3KB | ~400B |
| Ed25519 verify | ~8KB | ~800B |
| CRC32 (テーブル) | ~1KB | ~16B |
| **合計** | **~14KB** | **~1.4KB (ピーク)** |

## アプリ署名検証フロー

### ビルド時（ホスト PC）

```
1. アプリバイナリをビルド
2. AppHeader の signature フィールドをゼロクリア
3. ヘッダ + ペイロード全体の SHA-512 ハッシュを計算
4. Ed25519 秘密鍵でハッシュに署名 → signature[64] に格納
5. CRC32 を計算 → crc32 フィールドに格納
```

### ロード時（デバイス: Loader）

```
1. AppHeader 読み取り
2. magic / abi_version 検証
3. CRC32 検証（高速な完全性チェック）
4. target == RELEASE の場合:
   a. signature を一時保存し、ヘッダ上をゼロクリア
   b. ヘッダ + ペイロードの SHA-512 ハッシュを計算
   c. 公開鍵で Ed25519 署名検証
   d. signature を復元
   e. 検証失敗 → ロード拒否、Shell 有効化
5. MPU 設定 → アプリ起動
```

### CRC32 と Ed25519 の役割分担

| | CRC32 | Ed25519 |
|---|---|---|
| 目的 | 転送エラー検出 | 改竄検出 |
| 速度 | 非常に高速 | 比較的遅い（数十 ms） |
| 適用 | 常時（全ターゲット） | RELEASE のみ |
| 攻撃耐性 | なし（意図的改竄に無力） | 暗号学的安全性 |

## OTA/DFU 検証

Updater の DFU フローにおける検証:

```
FW_BEGIN  → サイズ、期待 CRC を受信
FW_DATA   → チャンクごとに CRC を累積計算
FW_VERIFY → 累積 CRC と期待値を比較
            → RELEASE の場合は Ed25519 署名も検証
FW_COMMIT → 検証成功後にメタデータを更新（確定）
```

FW_VERIFY で失敗した場合、書き込み済みデータは確定されず、次回起動時に前バージョンが使用される。

## 鍵管理

### 公開鍵の格納

- Ed25519 公開鍵（32B）はカーネルの Flash（.rodata）に埋め込む
- カーネル更新時にのみ公開鍵を変更可能
- アプリからはアクセス不可（カーネル Flash は MPU で保護）

### 鍵ペアの運用

| 鍵 | 格納場所 | 管理 |
|----|---------|------|
| 秘密鍵 | ビルドサーバー / 開発者 PC | オフライン保管推奨 |
| 公開鍵 | カーネル Flash (.rodata) | カーネルビルド時に埋め込み |

### 鍵更新

公開鍵の更新はカーネル自体の更新を伴う。
カーネル更新はブートローダー（umiboot）が担当し、本章のスコープ外。

---

## 組み込み制約

### リアルタイム安全性

- 暗号処理は **SystemTask でのみ実行**する
- AudioTask / ControlTask から呼ぶことはない
- Ed25519 検証は起動時の 1 回のみ（数十 ms のブロッキングは許容）

### スタック使用量

Ed25519 検証は ~800B のスタックを使用する。SystemTask のスタックサイズはこれを考慮して設計する。
