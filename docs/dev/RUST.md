# 技術選定レポート: 電子楽器開発における Rust vs C++
**― 構造的安全性と並行処理アーキテクチャの観点から ―**

## 1. はじめに
本ドキュメントは、次世代のシンセサイザーおよびDSPアプリケーション開発におけるプログラミング言語選定の判断材料を提供することを目的とする。

従来の業界標準である **C++** と、近年採用が進む **Rust** を比較し、特に「ポリフォニック（多音）アーキテクチャのメモリ効率」および「リアルタイムスレッドにおける安全性」の観点から分析を行う。

**結論の要約:**
C++は最大限の自由度を提供するが、その安全性はプログラマの規律に依存する。対してRustは、**DSPプログラミングにおける「正しい構造（データと状態の分離、スレッド間排他）」をコンパイル時に強制する**ことで、開発後半のデバッグコストを劇的に削減し、堅牢な製品品質を担保する。

## 2. アーキテクチャ比較：8和音シンセサイザーの実装

ポリフォニックシンセサイザーにおいて、「全ボイスで共通のパラメータ（Cutoff, Resonance等）」と「ボイスごとの内部状態（遅延バッファ, エンベロープ現在値）」をどう管理するかは、パフォーマンスとメンテナンス性に直結する。

### 2.1 C++のアプローチ：OOPによるカプセル化の弊害
C++における直感的なオブジェクト指向設計（OOP）では、係数と状態を一つのクラスにまとめる傾向がある。

```cpp
// C++: 係数と状態が混在した設計
class Filter {
    // 係数 (本来は全ボイスで共有すべき値)
    float a0, a1, b1, b2; 
    // 状態 (ボイスごとに異なる値)
    float z1, z2; 

public:
    void process(float input);
    void setCutoff(float freq); // 係数再計算
};

// 8ボイス分のインスタンス（係数も8回コピーされる）
std::vector<Filter> voices(8);
```

**技術的リスク:**
1.  **メモリ冗長性とキャッシュ効率:** 同じ係数データを各ボイスが個別に持つため、L1キャッシュを圧迫する。
2.  **更新の同期ズレ:** パラメータ変更時、8回ループして `setCutoff` を呼ぶ必要があり、計算コスト増大と位相ズレのリスクがある。
3.  **ポインタ共有の危険性:** メモリ節約のために係数を `Filter*` 等で共有しようとすると、ダングリングポインタ（解放済み領域へのアクセス）のリスクが発生する。

### 2.2 Rustのアプローチ：所有権システムによる強制的な分離
Rustの借用（Borrowing）ルールは、**「共有される不変データ」**と**「占有される可変状態」**の分離を設計レベルで強制する。

```rust
// 1. 係数 (Immutable / Shared): ヒープに1つだけ確保
struct FilterCoeffs {
    a0: f32, a1: f32, b1: f32, b2: f32,
}

// 2. 状態 (Mutable / Owned): ボイスごとに確保
struct FilterState {
    z1: f32, z2: f32,
}

// 3. シンセサイザー本体
struct Synthesizer {
    // Arc (Atomic Reference Count) でスレッドセーフに共有
    coeffs: Arc<FilterCoeffs>, 
    voices: [FilterState; 8],
}
```

**Rustの優位性:**
1.  **ゼロコストの安全性:** `Arc<T>` により、係数データは全ボイスで物理的に共有される。参照カウントにより、使用中にメモリが解放されることは**絶対にない**。
2.  **アトミックな更新:** `coeffs` のポインタ（Arc）をすげ替えるだけで、全ボイスの音色が瞬時に、かつ完全に同期して切り替わる。
3.  **書き込み事故の防止:** DSP処理中（`process`関数内）において、係数 `coeffs` は不変参照（`&T`）として渡されるため、誤って係数を書き換えるコードはコンパイルエラーとなる。

## 3. 並行処理の課題：Data Race と Race Condition

オーディオタスク（高優先度・ロック禁止）とUIタスク（低優先度）が通信する際、2種類の「競合」を区別する必要がある。

### 3.1 定義と挙動の違い

| 特徴 | データ競合 (Data Race) | 競合状態 (Race Condition) |
| :--- | :--- | :--- |
| **定義** | 複数のスレッドが同期なしに同一メモリにアクセスし、少なくとも1つが書き込みを行う状態。 | スレッド間の処理タイミングや順序に依存して、システムが意図しない状態になること。 |
| **現象** | **メモリ破壊、未定義動作。**<br>ビット列が破壊され、NaNや異常値が発生する。 | **論理的不整合。**<br>メモリは正常だが、計算結果が狂う（フィルタ発振など）。 |
| **C++** | **発生する。**<br>`std::atomic` の使用を忘れると即座に発生。コンパイラは検知しない。 | **発生する。**<br>プログラマのアルゴリズム設計能力に依存。 |
| **Rust** | **発生しない（コンパイルエラー）。**<br>`Send/Sync` トレイトにより、安全でない共有を物理的に拒否する。 | **発生しうる。**<br>ただし、メモリ破壊がないため原因特定が容易。 |

### 3.2 現場での影響：フィルタ係数更新のシナリオ

UIスレッドがフィルタ係数（`Cutoff`, `Resonance`）を更新中に、DSP割り込みが発生した場合：

* **Rustにおける貢献:**
    * データ競合（ティアリング）をコンパイル時に排除する。
    * 競合状態（新旧係数の混在による発振）を防ぐための「ダブルバッファリング」や「コマンドキュー」を実装する際、**ポインタの管理ミスによるクラッシュを恐れずに**複雑なロジックを記述できる。

## 4. 複雑性のスケーリング：C++が破綻する境界線

単体のモジュールではなく、相互参照や動的な生成・削除が発生する複雑なグラフ構造において、言語の差は決定的なものとなる。

### 4.1 モジュレーション・マトリクス (Dangling Pointer)

LFOやエンベロープ（Source）が、任意のパラメータ（Destination）を変調する多対多の接続。

* **課題:** ユーザー操作により「LFO 1」が削除された際、DSPスレッドがそのポインタを参照し続けているとクラッシュする。
* **C++のリスク:** 生ポインタ `Lfo*` で接続を管理しがちである。削除通知の同期をミスすると、即座に **Use-After-Free**（解放済みメモリ使用）となる。
* **Rustの強制力:** ライフタイムの不明確なポインタ保持を許さないため、**「IDベースの管理（Generational Arena）」**の実装を強く誘導する。
    * ポインタではなく整数ID（`LfoId`）で管理することで、実体が削除されていれば単に `None` を返し、安全に処理をスキップする構造になる。

### 4.2 エフェクト・チェインの動的組み換え (Iterator Invalidation)

演奏中にエフェクトの順序（例: Delay → Reverb）を入れ替える機能。

* **課題:** DSPスレッドが `std::vector` をループしている最中に、UIスレッドがそのベクタを書き換える（並べ替え/削除）。
* **C++のリスク:** **イテレータの無効化**が発生し、ループ処理が不正なアドレスに飛び、メモリ破壊や無限ループを引き起こす。ロック（Mutex）は音切れの原因となるため使いづらい。
* **Rustの強制力:** 借用規則により、読み取り中（DSP）のベクタに対して書き込み（UI）を行うコードはコンパイルできない。
    * これにより、**「ダブルバッファリング（グラフの二重化）」**や**「メッセージパッシング」**といった、ロックフリーかつ安全な更新パターンの採用が不可避となる。

## 5. 総合結論

電子楽器アプリケーション開発において、Rustを選択する最大のメリットは「安全性」という受動的な言葉に留まらない。

Rustのコンパイラは、DSPプログラミングにおけるベストプラクティスである**「データと状態の分離」**や**「ロックフリーなデータ構造」**を、**実装必須の要件**として開発者に突きつける。

* **C++:** 熟練者が注意深く書けば、Rustと同等の品質を出せる。しかし、チーム開発や仕様変更の圧力の中で、その規律を維持するコストは指数関数的に増大する。
* **Rust:** コンパイルが通る＝**「メモリ破壊やデータ競合が存在しない」**ことが保証される。開発者は「音が出るかどうか」「論理が正しいか」という本質的な課題のみに集中できる。

以上の理由により、高い信頼性と長期的な保守性が求められるオーディオ・エンジン開発において、RustはC++に代わる最も合理的な選択肢であると結論付ける。
