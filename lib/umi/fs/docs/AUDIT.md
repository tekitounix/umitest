# リファレンス実装 解析レポート

FATfs R0.16 および littlefs v2.11 のリファレンスC実装を解析し、
クリーンルーム再実装で克服すべき問題点を特定した。

---

## FATfs R0.16

対象: `.refs/fatfs/source/ff.c`

### 評価サマリ

| 観点 | 評価 |
|------|------|
| 電源遮断耐性 | 不十分（FATフォーマットの構造的限界） |
| エラー伝搬 | 一部欠落（2nd FAT, FSInfo書き込みエラー無視） |
| 破損データ耐性 | 不十分（循環クラスタチェーンで無限ループ） |
| 並行安全性 | 条件付き（FS_REENTRANT=1でmutex枠組みあり） |
| コード品質 | 良好（成熟したコードベース） |

### 問題一覧

#### Critical

**F1: f_rename のクロスリンク窓**

新エントリ作成後、旧エントリ削除前に電源断が発生すると、
同一クラスタチェーンが2つのディレクトリエントリから参照される。
データ破壊が波及する不可逆状態。

コード自身が `/* Start of critical section where an interruption can cause a cross-link */` と認識している。

**F2: グローバル可変状態**

`FatFs[]`, `Fsid`, `CurrVol`, `LfnBuf` 等がグローバル配列。
複数ボリュームで`Files[]`が競合する可能性がある。

#### High

**F3: 2nd FAT / FSInfo の書き込みエラー無視**

`disk_write`の戻り値が完全に無視される。冗長性が黙って失われる。

**F4: 循環クラスタチェーンで無限ループ**

`remove_chain`/`f_lseek`のFATチェーン走査に循環検出がない。
破損メディアでプログラムが永久停止する。

#### Medium

**F5: f_truncate のサイズ不整合**

`remove_chain`がエラーを返した後も`objsize`を更新する。
次回syncで不整合がディスクに永続化される。

**F6: ff_memalloc の使用**

`FF_USE_LFN == 3`でヒープ割り当てが発生する。
組込み環境ではヒープ枯渇やフラグメンテーションの原因になる。

**F7: f_unlink でロストクラスタ**

ディレクトリエントリ削除済みだがクラスタチェーンが割当済のまま残る。
繰り返し発生で最終的にディスクフルになる。

**F8: FAT二重化の非アトミック更新**

1st FAT書き込み成功後、2nd FAT書き込み前に電源断でFAT不整合。

### 電源遮断シナリオ

| シナリオ | 結果 | 復旧 |
|---------|------|------|
| FAT更新中に遮断 | FAT不整合 | fsck必要 |
| データ書き込み中に遮断 | 部分書き込み、サイズ不整合 | fsck必要 |
| ディレクトリ更新中に遮断 | クラスタリーク | fsck必要 |
| f_rename 中に遮断 | クロスリンク（Critical） | 手動修復 |

## littlefs v2.11

対象: `.refs/littlefs/lfs.c`

### 評価サマリ

| 観点 | 評価 |
|------|------|
| 電源遮断耐性 | 堅牢（COW + CRC + gstateで自動復旧） |
| エラー伝搬 | 良好（一貫してエラーコードを伝搬） |
| 破損データ耐性 | 良好（CRC検証で自動検出・棄却） |
| 並行安全性 | 条件付き（LFS_THREADSAFEでロック枠組み） |
| コード品質 | 良好（設計が明確） |

### 電源遮断耐性の設計

littlefs が保証するのは**一貫性（consistency）**であり、**耐久性（durability）**ではない。

**3層防御メカニズム:**

1. **メタデータペアのCOW**: 各ディレクトリは2ブロックのペア。更新は一方への追記で、CRC付きコミットマーカーで確定。電源断時はCRC不正なエントリを無視し、直前の整合状態に自動復帰
2. **CRCによるコミット境界**: 各コミットにCRC-32付与。FCRCでコミット後の消去済み領域も検証
3. **gstateによる分散トランザクション**: 2ディレクトリにまたがる操作はgstateで管理。マウント時に不完全な操作を自動削除

### 問題一覧

#### High

**L1: ASSERT 70箇所がリリースで無効**

`LFS_NO_ASSERT`定義時、全てのASSERTが無効化される。
破損メタデータで未定義動作やスタックバッファオーバーフローの可能性がある。

**L2: lfs_dir_traverse の再帰深度制限がASSERTのみ**

`LFS_NO_ASSERT`ビルドでは深度制限が機能しない。
破損メタデータでスタックオーバーフローが発生する。

**L4: lock/unlock コールバック未検証**

`LFS_THREADSAFE`有効でコールバック未設定の場合、NULLデリファレンス。

#### Medium

**L3: lfs_npw2 の未定義動作**

`a == 1`で`__builtin_clz(0)`が呼ばれる。
現在の呼び出しパスでは`a >= 2`が保証されているが、防御的修正が望ましい。

**L5: rcache バッファの暗黙的再利用**

`rcache.buffer`をtruncate時に一時バッファとして再利用している。
内部変更で壊れる可能性がある。

### ウェアレベリング

- 動的ウェアレベリング: `lfs->seed`からランダムに開始位置を決定
- メタデータリロケーション: `block_cycles`設定で強制リロケーション間隔を制御
- 静的ウェアレベリングは行わない（設計上のトレードオフ）

### ブロックデバイス層の前提

littlefs の電源遮断耐性はブロックデバイス層の品質に依存する:
- `prog`は物理的なプログラム完了を保証すること
- `sync`は物理媒体への書き込み完了を保証すること
- キャッシュを持つデバイスでは`sync`がキャッシュフラッシュを含むこと

## 比較

| 観点 | littlefs | FATfs |
|------|----------|-------|
| アトミック書き込み | CRC検証付きコミット | セクタ単位（非アトミック） |
| メタデータ冗長化 | デュアルペア（自動復旧） | FAT二重化（非アトミック） |
| 電源遮断復旧 | mount()で自動 | fsckツール必要 |
| データ検証 | 全読み込みでCRC | なし |
| ウェアレベリング | ブロックリロケーション | なし |
| 割り込み安全性 | 読み込み安全（CRC検証） | 安全でない |

**選定指針:**
- 信頼性が最重要 → littlefs
- FAT互換性が必要 → FATfs（mutex排他制御 + sync()頻繁呼び出し必須）
- いずれもISRからの直接操作は禁止
